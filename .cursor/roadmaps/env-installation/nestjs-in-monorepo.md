# Додавання NestJS до монорепо (Turborepo + pnpm)

Roadmap за узгодженим у чаті флоу: окремий застосунок **`apps/api`**, без зайвого переносу з «тимчасової» теки, з **pnpm** і **Nest CLI**.

---

## Передумови

- У корені репо: **pnpm**, **Turborepo**, **`pnpm-workspace.yaml`** з покриттям `apps/*` (або явно `apps/api`).
- Встановлені **Node.js** (офіційна документація Nest зазвичай вимагає **Node ≥ 20**; у проєкті може бути `engines` — звіритися).
- Глобально або через `pnpm dlx` доступний **Nest CLI** (`@nestjs/cli`).

---

## Крок 1. Місце в дереві проєкту

- Цільовий шлях: **`apps/api`** — окремий Nest-застосунок у монорепо.
- Не дублювати Nest у кількох `apps/*`, якщо не планується кілька API.

---

## Крок 2. Створити застосунок через Nest CLI у `apps/api`

### 2.1. Порожня тека

1. Створити каталог **`apps/api`**.
2. Переконатися, що він **порожній** (немає `package.json`), якщо використовуєте варіант **`nest new .`**.

### 2.2. Команда генерації

З теки **`apps/api`**:

```bash
nest new . --package-manager pnpm --skip-git --strict
```

**Параметри:**

| Параметр | Навіщо |
|----------|--------|
| `.` | Скаффолд у поточній теці, без зайвої вкладеності `nest-app/nest-app`. |
| `--package-manager pnpm` | Єдиний менеджер пакетів з монорепо. |
| `--skip-git` | Не ініціалізувати окремий git у `apps/api` (репозиторій один — корінь). |
| `--strict` | Суворіший TypeScript (рекомендація Nest / First steps). |

**Альтернатива** (менш зручна): `nest new <ім’я>` поза `apps/api`, потім перенести вміст у `apps/api` і підчистити шляхи.

### 2.3. Офіційне джерело

Загальна ідея збігається з [First steps — NestJS](https://docs.nestjs.com/first-steps): `nest new` + TypeScript; для монорепо додатково важливі прапорці **pnpm** та **skip-git**.

---

## Крок 3. Залежності з корня workspace

З **кореня** монорепо:

```bash
pnpm install
```

Щоб lockfile і `node_modules` були узгоджені для всіх workspace-пакетів, включно з `apps/api`.

---

## Крок 4. Інтеграція з Turborepo та спільними конфігами

Після генерації CLI проєкт «ізольований». Його варто **вписати** в існуючі правила монорепо.

### 4.1. Скрипти в `apps/api/package.json`

Рекомендовано додати / перевірити:

- **`dev`** — наприклад `nest start --watch`, щоб **`turbo run dev`** міг піднімати API разом з фронтами.
- **`check-types`** — наприклад `tsc --noEmit` (або еквівалент), щоб кореневий **`pnpm check-types`** охоплював API.

Стандартні з CLI зазвичай вже є: **`build`** (`nest build`), **`lint`**, **`start`**, **`start:dev`** тощо.

### 4.2. TypeScript і `@repo/typescript-config`

- У **`apps/api/tsconfig.json`** (або через окремий `tsconfig.build.json`, залежно від прийнятої схеми) узгодити з **`@repo/typescript-config`**: наприклад `extends` на спільний базовий конфіг монорепо.
- За потреби додати явний **`include`** (наприклад `src`), якщо IDE або `tsc` без `nest build` поводяться неочікувано.

### 4.3. `turbo.json`

- Додати до **`build.outputs`** артефакти Nest: **`dist/**`** (поряд із `.next/**` для Next), щоб кеш Turbo для API був коректний.
- Переконатися, що для `apps/api` визначені ті самі «дієві» таски, що й для інших apps: **`dev`**, **`build`**, **`lint`**, **`check-types`** (як у домовленостях монорепо).

### 4.4. Порт за замовчуванням

У шаблоні **`main.ts`** часто:

`await app.listen(process.env.PORT ?? 3000);`

У тому ж монорепо **Next** зазвичай теж **3000** → конфлікт при одночасному запуску.

**Рекомендація:** для API задати інший дефолт (наприклад **3001**) або завжди виставляти **`PORT`** у `.env` / скриптах лише для `apps/api`.

---

## Крок 5. Перевірка

1. З корня: **`pnpm install`**.
2. Запуск API: **`pnpm --filter api start:dev`** або через **`turbo run dev`** (якщо налаштовано `dev` у `api`).
3. Збірка: **`pnpm build`** / **`turbo run build`** — `apps/api` має збиратися в **`dist/`**.
4. За наявності: **`pnpm lint`**, **`pnpm check-types`** — API у складі workspace.
